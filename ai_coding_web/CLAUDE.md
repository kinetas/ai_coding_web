# Claude.md

## 프로젝트 한줄 정의
공공 데이터와 뉴스 데이터를 ETL로 수집·정제해 저장하고, 사용자가 `My Page`에서 개인화된 차트와 워드클라우드를 구성해 보는 데이터 시각화 서비스입니다.

## 목표
- 사용자가 복잡한 데이터 수집 과정을 몰라도 바로 의미 있는 인사이트를 볼 수 있게 한다.
- ETL → 저장 → 시각화 흐름을 안정적으로 운영한다.
- MVP의 속도와 운영 효율을 유지하면서, 이후 확장 가능한 구조를 만든다.

## 현재 구조 요약

### 기술 스택
- Backend: FastAPI, SQLAlchemy
- Database: SQLite(dev) / PostgreSQL(prod)
- Frontend: Vanilla JS, Static HTML
- ETL: `etl.py`, `crawler/news_pipeline.py`, 공공데이터 API, Google News RSS

### 아키텍처
- 패턴: Controllers → Services → Repositories
- 저장소 전환: `CONTENT_SOURCE=local|supabase`
- 핵심 특징:
  - 사용자 대시보드용 데이터는 서버 저장형
  - ETL과 사용자 API가 분리된 구조
  - 운영 환경과 개발 환경 인증 방식이 구분됨

### 인증
- Prod: Supabase JWT Bearer
- Dev: SQLite 세션 쿠키
- ETL 보호: `X-ETL-Token`
- 보호 경로:
  - `/api/builder/*`
  - `/api/ingest/*`

## 핵심 사용자 경험
- 공개 페이지에서 카테고리별 추이와 농산물 분석을 바로 확인할 수 있어야 한다.
- 로그인 사용자는 `My Page`에서 자신만의 대시보드를 구성할 수 있어야 한다.
- 중요한 것은 "예쁜 시각화"보다 "맥락이 읽히는 시각화"다.

## 현재 강점
- ETL부터 시각화까지 파이프라인이 명확하다.
- MVP로 빠르게 검증하기 좋은 구조다.
- `CONTENT_SOURCE`로 저장소 구현을 유연하게 교체할 수 있다.
- Static HTML + FastAPI 조합은 비용 효율이 높다.
- 워드클라우드 생성 조건과 재시도 규칙이 이미 정의되어 있어 운영 기준이 분명하다.

## 현재 한계와 핵심 리스크

### UX
- 사용자가 빈 상태에서 직접 차트와 워드클라우드를 구성해야 해 초기 진입 장벽이 있다.
- 데이터 부족이나 ETL 진행 상황이 사용자에게 충분히 설명되지 않을 수 있다.
- 모바일에서 대시보드 사용성이 깨질 가능성이 있다.

### 비즈니스
- 공공 데이터 API와 Google News RSS 정책 변화에 의존한다.
- 사용자 리텐션을 높일 반복 방문 동기가 아직 약하다.

### 시스템
- ETL 재시도는 있으나 네트워크 타임아웃, 레이트 리밋 대응은 더 정교해질 필요가 있다.
- 프론트가 커질수록 Vanilla JS 유지보수 비용이 급격히 늘 수 있다.

### 데이터 품질
- 뉴스 특성상 불용어와 노이즈 제거 품질이 결과물의 수준을 크게 좌우한다.
- 데이터 최신성 표기가 약하면 분석 신뢰도가 낮아질 수 있다.

## 우선순위 제안

### 1. 바로 효과가 큰 개선
- 기본 템플릿 제공:
  - `농산물 가격 추이 템플릿`
  - `주간 뉴스 키워드 템플릿`
  - `카테고리별 이슈 변화 템플릿`
- 데이터 상태 메시지 개선:
  - `데이터 없음` 대신 `수집 중 (10/15 단어)` 같은 진행형 피드백 제공
- `Last updated` 표시 추가
- 모바일 반응형 대시보드 검증

### 2. 안정성 개선
- ETL에 지수 백오프와 타임아웃 정책 추가
- 외부 소스 실패 시 부분 실패 허용 및 기존 데이터 유지 전략 명확화
- 비동기 DB IO 사용 여부와 병목 구간 점검

### 3. 데이터 품질 개선
- 카테고리별 특수 불용어 사전 강화
- 의미 없는 뉴스 토큰 제거 품질 향상
- 워드클라우드와 뉴스 분석 결과의 품질 기준 수치화

### 4. 제품 가치 확장
- 가격 급변 시점과 뉴스 키워드의 상관관계 분석
- 특정 키워드/품목 기준 알림 기능
- 반복 방문을 유도하는 개인화 피드 추가

## 제품 원칙
- 사용자는 설정이 아니라 결과를 먼저 봐야 한다.
- 데이터가 없을 때도 "왜 없는지"를 설명해야 한다.
- 실시간보다 중요한 것은 신뢰 가능한 준실시간 데이터다.
- 기능 추가보다 데이터 품질과 사용성 개선이 우선이다.
- 확장은 가능하되 MVP의 단순함은 유지한다.

## 운영 체크포인트
- 외부 데이터 공급원 장애 시 복구 가능 여부
- ETL 실행 시간과 실패율
- 워드클라우드 생성 성공률
- 사용자 템플릿 사용률
- 모바일 대시보드 이탈률
- 데이터 최신성 표시 정확도

## 추천 로드맵

### 단기
- 템플릿 추가
- 진행 상태 메시지 추가
- 최신 갱신 시각 표시
- ETL 백오프 적용

### 중기
- 불용어 사전 고도화
- 이상치와 뉴스 연관 분석
- 프론트엔드 상태 관리 최소 규칙 정립

### 장기
- 키워드/품목 알림
- 구독형 리텐션 기능
- 고도화된 인사이트 추천 대시보드

## 결론
이 프로젝트는 구조적으로 이미 MVP에 매우 적합하며, ETL·저장·시각화 흐름도 명확합니다. 가장 큰 개선 포인트는 새로운 대형 기능 추가보다, 템플릿 기반 UX, 데이터 상태 가시성, ETL 안정성, 불용어 중심의 데이터 품질 향상입니다. 이 네 가지를 먼저 강화하면 서비스 완성도와 실사용 가치는 크게 올라갑니다.
